# スター自動確認システム（Future構想）

## 1. 概要（Overview）

- SNS上の公開情報を活用してスターアカウントが「本物っぽく見えるか」を自動チェックし、LEVEL1認証の精度を高める仕組み。
- 目的は「なりすましリスクの低減」と「運営負荷の最小化」を両立させ、明らかな偽アカウントを早期検知しつつ本物に対してスムースな認証体験を提供すること。

## 2. 対象SNSと方針

- **YouTube / Instagram**
  - 公式API（YouTube Data API / Instagram Graph API）を使って自動スクリーニング。
  - 取得する指標としてアカウント年齢、投稿数、フォロワー数、直近投稿のいいね数・コメント数などを想定。
  - APIコストは最初は無料枠・クォータ制限内で運用し、必要があればリクエストタイミングを分散。

- **X（旧Twitter）**
  - APIコスト・ポリシーの制約が厳しいため、MVPでは自動スクリーニングから除外。
  - 当面は「共有された固定ポスト（認証コード）をスターに掲載」→ 運営が目視で確認するフローを予定。
  - 将来的にAPIアクセスが安定すれば、自動判定ロジックを追加するオプションとして残す。

## 3. 自動取得する指標（Metrics）

- 共通の考え方：公開データから「運用されている実在アカウントか」「極端に不自然でないか」を掴むために最低限の指標を取得。
- 例（YouTube / Instagram）：
  - アカウント開設日（アカウント年齢）
  - 総投稿数 / 直近N件の投稿数（活発度）
  - フォロワー数・フォロー数（信頼性の指標）
  - 直近投稿の平均いいね数・コメント数（エンゲージメント）
- 投稿内容の自動解析はMVP段階では実施せず、「量と基本的なエンゲージメント」のデータに集中。
- APIエラー・非公開アカウントなど情報が取得できないケースは無理にPASSせず `manual_review` に回す想定。
  - 特に鍵アカ（private）については自動判定を行わず、運営が目視で確認する運用へ：
    - STARLIST公式アカウントでフォローリクエストに対応いただき、投稿やフォロワーを確認。
    - 承認されない/確認できない場合は LEVEL1 ではなく LEVEL0 + 収益制限のままに据置。

## 4. 判定ロジック（Heuristics v1）

- 最初のバージョンは「ざっくりしたしきい値」を用いて経験に応じて調整。
- 例（仮の数値）：
  - 投稿数：合計10件以上、直近3件以上の投稿がある
  - フォロワー数：100人以上
  - 直近投稿の平均いいね数：一定以上
  - APIレスポンスで指標が欠けていた場合は、安易に `passed` にしない
- 判定パターン：
  - 条件クリア → `passed` → LEVEL1付与候補としてマーク
  - グレーゾーン/情報不足 → `manual_review` → 運営が目視で確認
  - 明らかに不自然（極端なスパム/急激成長など） → `failed` → LEVEL0維持＋課金制限を強化案

## 5. なりすまし疑義時の考え方（High-level Policy）

- 通報や運営監視で「なりすまし疑義」が出た場合、以下のフローを想定：
 1. 対象スターを一時凍結（新規課金停止、アカウント自動更新停止）
 2. 本人を名乗る側と現在のスター側の両方から追加証拠を収集
 3. 偽と判明した場合：
    - 該当アカウントをBAN
    - 既存課金に対して返金またはクレジット付与でファンを救済（決済ポリシーと連動）
 4. 真偽不明の場合：
    - LEVEL0に格下げし、課金や収益関連機能を制限
    - 「本人保証はできないがプラットフォームとして収益活動を制限する」という体制を維持
- 詳細なタイムライン・責任者は別途 `IMPERSONATION_HANDLING_POLICY` 系ドキュメントで掘り下げる前提。

## 6. v1実装スコープのメモ

- 本ドキュメントは「Future構想」のメモであり即実装を意味しない。
- ざっくり想定されるv1スコープ：
  - `star_verifications` のような新規テーブル（プラットフォーム／ステータス／取得指標／チェック日時を保持）
  - YouTube / Instagram 向けの自動チェック処理（Supabase Edge Functions / cron など）
  - 「要手動レビュー」スターを一覧表示する運営用ダッシュボードの簡易版
  - スター詳細・決済画面に認証バッジ＋注意書きの表示ロジック
  - `manual_review` と `failed` のトラッキングや通知ルート（Slack/メール）を整備

今後の要件固めやプロトタイプでは、このメモをベースに詳細化を進める。
