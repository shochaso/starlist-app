# STARLIST 優先順位マトリックス

## 背景

STARLISTプロジェクトのタスク・機能開発における優先順位付けを体系的に管理するためのフレームワークです。本マトリックスはビジネス価値と技術的実現性の観点から、全ての取り組み事項の優先度を決定します。

## 要件

### 機能要件
- [x] 2軸（ビジネス価値 × 技術的実現性）での優先順位評価
- [x] 各タスクのスコアリングとランク付け
- [x] 定期的な見直しと更新メカニズム
- [x] 意思決定の透明性確保

### 非機能要件
- [x] 客観的な評価基準の適用
- [x] ステークホルダーの合意形成
- [x] 定期的な見直し（月次）

## 評価軸と基準

### X軸: ビジネス価値 (Business Value)
1. **高 (8-10)**: 収益直接貢献、競争優位性確保、規制対応
2. **中 (4-7)**: ユーザー体験改善、運用効率化、技術的優位性
3. **低 (1-3)**: 将来的な価値、技術的興味、マイナー改善

### Y軸: 技術的実現性 (Technical Feasibility)
1. **高 (8-10)**: 既存技術スタック、短期間実装、既知のリスク
2. **中 (4-7)**: 一部新規技術、適度な期間、コントロール可能なリスク
3. **低 (1-3)**: 新規技術スタック、長期間実装、高い不確実性

## 評価プロセス

### 1. 初期評価フェーズ
- **ビジネス価値評価**: Product Team × ステークホルダー
- **技術的実現性評価**: Engineering Team × アーキテクト
- **スコアリング**: 1-10の定量評価

### 2. 合意形成フェーズ
- **クロスチェック**: 評価結果の相互検証
- **調整会議**: 相違点の解消と合意形成
- **最終決定**: マトリックスへの配置

### 3. 実施計画フェーズ
- **リソース配分**: 優先度に基づくチーム割当
- **スケジューリング**: 依存関係考慮の実施計画
- **リスク評価**: 実施に伴うリスクの特定

## 定期見直しサイクル

### 月次レビュー
- **完了タスクの除外**: 完了した項目のマトリックスからの削除
- **新規タスクの追加**: 新しい要求事項の評価・追加
- **優先度再評価**: 状況変化に伴う再スコアリング

### 四半期レビュー
- **戦略的見直し**: 事業目標との整合性確認
- **技術トレンド反映**: 新技術の影響評価
- **リソース最適化**: チーム能力に応じた優先順位調整

## DoD (Definition of Done)

### 優先順位付け完了の条件
- [x] 全タスクの評価完了
- [x] ステークホルダーの合意取得
- [x] マトリックスへの正確な配置
- [x] 実施計画の策定
- [x] 定期見直し計画の設定

## Runbook

### 新規タスク評価手順
1. **タスク登録**: 新しい取り組み事項の特定・登録
2. **評価実施**: 2軸でのスコアリング
3. **合意形成**: 関係者間での調整
4. **マトリックス配置**: 優先度に基づく位置決定
5. **実施計画**: リソース・スケジュールの確定

### 緊急対応プロセス
- **緊急度評価**: 通常評価プロセスのスキップ判断
- **暫定優先度**: 暫定的な高優先度配置
- **後日精査**: 通常プロセスでの再評価

## リスク

### 評価バイアスのリスク
- **主観的評価**: 評価者のバイアスによる歪み
  - **対策**: 複数評価者による相互検証

### 状況変化のリスク
- **優先度の陳腐化**: 事業環境の変化による優先順位変更
  - **対策**: 定期的な見直しサイクルの厳守
