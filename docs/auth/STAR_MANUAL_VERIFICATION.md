# スター手動本人確認フロー仕様（LEVEL1 運用）

## 1. 目的・スコープ

- **目的**  
  スターリスト上の「スター」アカウントに対して運営が手動でSNS情報を確認し、LEVEL1（SNS連携済み）認証の付与／却下をルール化する。
- **スコープ**  
  現時点ではAPIによる自動チェックを使わず、すべて検証官（運営）がブラウザでSNSを目視確認する前提。対象はスターが申請した YouTube / Instagram / X（旧Twitter）など。

## 2. 用語・認証レベル

- **LEVEL0: 未認証スター**  
  メール認証＋ログイン済みだがSNS本人性確認は未実施。課金機能には上限/制限がかかる（別仕様参照）。
- **LEVEL1: SNS連携済み（手動確認済み）スター**  
  運営がSNSを目視確認し、「本人と見なしてよい」と判断した状態。自動化が整うまでの暫定・公式ステータス。
- **なりすまし疑義**  
  本人または第三者から「このスターは偽では？」という報告が入ったモード。

## 3. 手動確認が走るトリガー

1. **スターからの認証申請**  
   管理画面の「LEVEL1認証申請」＋公式SNS URL提出時に手動確認キューへ登録。
2. **新規登録スターの自動キューイン（オプション）**  
   一定条件（フォロワー数・アクティブ度）を満たすスターは自動的に要確認キューに入れる運用も許容（将来仕様）。ただしこのドキュメントでは手動確認前提。
3. **なりすまし通報**  
   本人・ファン・モデレーションチームからの通報があれば即座に確認キューへ。

## 4. 手動確認フロー（時系列）

### 4.1 申請受付

- スターが申請フォームで提出する内容：  
  公式SNSのURL（YouTube / Instagram / X）＋任意の補足情報（事務所・別名義など）。
- バックエンドでは例として以下の状態更新を想定：  
  `verification_status = "pending_manual"`、`requested_at` に申請日時、`requested_by` にスターID。

### 4.2 運営側の確認作業

1. **キュー画面から確認する初期情報**  
   スター名・ユーザID・申請SNS URL・申請日時・直近の問い合わせ履歴など。
2. **SNSをブラウザで開いて以下チェック**：
   - **(a) プロフィール欄と認証コード**  
     - STARLISTが発行した認証コード（例: `SL-XXXX`）がプロフィール欄に反映されているか。未掲載の場合は本人確認の証拠不足。
     - プロフィール名・アイコン・紹介文がスター情報と整合しているか。
   - **(b) 投稿内容と反応（いいね・コメント）**  
     - 投稿が本人の活動と一致しているか、いいね/コメント数やコメント内容もざっくり確認。  
     - コメント欄がスパムやbotばかりでないか（同一文言ばかり・外国語スパムなど）をチェック。
   - **(c) フォロー／フォロワーの実態**  
     - フォロワー数・フォロー数のバランス、極端な相互フォロー（follow4follow）やスパムっぽいアカウントばかりでないかを確認。
   - **(d) 鍵アカウント（非公開）**  
     - 投稿/フォロワーを確認するため、STARLIST公式アカウントからのフォロー承認を依頼。  
     - 承認されない/確認できない場合は `verification_status = "need_more_info"` か `rejected` とし、LEVEL1は付与しない。
3. **必要に応じて追加質問**  
   - 例：相互フォロー多めの理由、他SNSとの関連性、事務所名など追加情報をチャットやメールで照会。

### 4.3 判定・登録

- **承認（LEVEL1付与）**  
  - SNSプロフィールコード・投稿・フォロワーの整合性により本人性が確認できた時。  
  - 例: `verification_level = 1`、`verification_status = "approved"`、`verified_at`/`verified_by` を記録。
- **保留（追加情報待ち）**  
  - 鍵アカ未承認・情報不足・コメント欄がグレー。  
  - `verification_status = "need_more_info"` とし、スターへ追加情報依頼を送付。
- **却下（LEVEL0維持）**  
  - 明らかなスパム/なりすまし・整合しない投稿。  
  - `verification_status = "rejected"`、内部メモに却下理由を残す。

## 5. SNSごとの確認ポイント（チェックリスト）

- **YouTube**  
  - チャンネル名・アイコンがスター情報と一致または強い関連。  
  - 概要欄に名前・リンク・事務所等があるか。  
  - 投稿本数・コメントの雰囲気（スパム多発でないか）。
- **Instagram**  
  - ユーザ名・表示名・アイコン。  
  - 投稿内容が活動と自然に一致しているか。  
  - 相互フォロー系やスパムばかりのフォロワーに注意。
- **X（旧Twitter）**  
  - 固定ポスト/プロフィールに認証コードを掲載してもらうと判断しやすい。  
  - 投稿の語り口・告知内容が本人らしいか。  
  - フォロワー・リプ欄のトーン（botが多くないか）を確認。

## 6. なりすまし疑義が出たときのフロー（簡易版）

1. **起点**  
   - 本人やファン・内部監視からの通報で疑義が発生。
2. **即時措置**  
   - 対象スターを一時停止（例: `account_status = "suspended_suspected_impersonation"`）。  
   - 新規課金停止・既存サブスクの自動更新をOFF。
3. **調査**  
   - 申請中のスターと「本物」を名乗る側の両方から証拠（DMやプロフィールのコードなど）を取得。  
   - 必要なら公式SNSにコード掲載やDMで本人確認を依頼。
4. **結果反映**  
   - **偽と判断** → アカウントBAN・返金／クレジット付与（決済ポリシー連携）。  
   - **真偽不明** → LEVEL0に格下げ・課金制限（本人保証せず、被害最小化優先）。

## 7. スター／ファンへの表示仕様（手動フェーズ）

- **スター側マイページ**  
  - 各ステータス（LEVEL0 / pending_manual / LEVEL1）で文言を分かりやすく表示。  
- **ファン向け表示**  
  - LEVEL0: 「SNS本人確認は未完了です」等の注意書き。  
  - LEVEL1: 「STARLIST運営がSNS本人確認済みです」と認証バッジ表示。  
  - なりすまし疑義中: 「本人性調査中のため新規課金停止中」と明記。

## 8. 将来の自動化との関係

- このドキュメントは自動確認導入までの暫定フロー。  
- 将来的にはYouTube/Instagram APIによる自動チェックを導入し、ここで定義したチェックポイント（プロフィールコード・投稿/反応・フォロー関係・鍵アカ対応）を Heuristics のベースにする予定。  
- `manual_review` 相当の判断は人力に残し、それ以外を自動化する移行シナリオを想定。

今後の運用マニュアル・自動化仕様の起点として活用してください。
